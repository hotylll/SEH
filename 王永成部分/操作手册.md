## 1 引言

### 1.1 编写目的

本手册面向**系统管理员**，说明系统的安装启动、数据源配置、采集任务管理、报表文件维护、审计日志查看以及日常运维操作。普通用户的查询与报表操作请参阅《用户手册》。

### 1.2 适用对象

| 角色 | 权限范围 |
|------|----------|
| 系统管理员（admin） | 全部配置、采集、用户、审计操作 |
| 运维人员 | 服务启停、健康检查、数据备份 |

### 1.3 术语

| 术语 | 说明 |
|------|------|
| 数据源 | 采集来源配置，存于 data_sources 表 |
| 采集任务 | 一次采集执行记录，存于 collect_tasks 表 |
| 审计日志 | 关键操作记录，存于 audit_logs 表 |
| endpoint | 数据源访问地址（URL 或文件路径） |

## 2 系统安装与启动

### 2.1 环境要求

**表2-1 服务器环境**

| 项目 | 要求 |
|------|------|
| 操作系统 | Windows 10+ / Linux |
| Python | 3.10 及以上 |
| 磁盘 | 至少 500MB 可用空间（含数据库与报表） |
| 端口 | 默认 8000（可自定义） |

### 2.2 安装步骤

1. 进入项目根目录 `SEH-main`。
2. 确认 Python 可用：

```powershell
python --version
```

3. 安装运行依赖：

```powershell
python -m pip install -r requirements.txt
```

4. 首次启动会自动创建 `data/app.db` 并写入演示数据。

### 2.3 启动服务

```powershell
python -m app.main --host 127.0.0.1 --port 8000
```

**表2-2 启动参数**

| 参数 | 默认值 | 说明 |
|------|--------|------|
| --host | 127.0.0.1 | 监听地址，生产环境可改为 0.0.0.0 |
| --port | 8000 | 监听端口 |
| --db | data/app.db | SQLite 数据库路径 |

### 2.4 启动验证

访问健康检查接口：

```powershell
curl http://127.0.0.1:8000/api/v1/health
```

**预期响应：**

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "status": "ok",
    "db": "data/app.db",
    "version": "1.0.0",
    "time": "2026-06-23T08:00:00+00:00"
  }
}
```

### 2.5 Docker 部署（可选）

```powershell
docker compose up -d
```

详见罗元恒编写的《系统部署与 CI/CD 方案》。

### 2.6 停止服务

- 前台运行：在终端按 `Ctrl+C`。
- Docker：`docker compose down`。

## 3 数据源配置

### 3.1 管理入口

管理员通过以下方式管理数据源：

1. **管理后台界面**（见《界面设计说明》「系统管理页」）— 推荐
2. **REST API** — 脚本批量配置或联调

### 3.2 支持的数据源类型

**表3-1 数据源类型**

| type | 名称 | endpoint 示例 |
|------|------|---------------|
| news | 新闻源 | `https://example.com/news` |
| forum | 论坛源 | `https://example.com/forum` |
| api | 接口源 | `file://data/samples/demo_api.json` |
| csv | 文件源 | `data/samples/demo_news.csv` |

详细字段规范见《数据源接入规范》。

### 3.3 新增数据源

**界面操作：**

1. 打开前端页面 → 「数据源管理」。
2. 点击「新增数据源」。
3. 填写表单：

**表3-2 新增数据源表单字段**

| 字段 | 必填 | 说明 |
|------|:----:|------|
| 名称 | 是 | 便于识别的显示名 |
| 类型 | 是 | news / forum / api / csv |
| endpoint | 是 | URL 或本地文件路径 |
| keywords | 否 | 逗号分隔，用于采集过滤 |
| schedule | 否 | manual（默认）/ hourly / daily / weekly |
| 状态 | 否 | enabled（默认）/ disabled |

4. 点击「保存」，系统校验通过后写入数据库。

**API 操作：**

```powershell
curl -X POST http://127.0.0.1:8000/api/v1/sources ^
  -H "Content-Type: application/json" ^
  -d "{\"name\":\"演示新闻源\",\"type\":\"news\",\"endpoint\":\"https://example.com/news\",\"keywords\":\"人工智能,新能源汽车\",\"schedule\":\"manual\"}"
```

**常见校验错误（HTTP 400）：**

| 错误信息 | 处理 |
|----------|------|
| endpoint 不能为空 | 填写有效地址 |
| news 类型 endpoint 应为 http(s) URL | 检查 URL 协议 |
| csv 文件不存在 | 确认文件路径正确 |
| source_type 必须是 [...] | 类型拼写错误 |

### 3.4 查询数据源列表

```powershell
curl "http://127.0.0.1:8000/api/v1/sources"
curl "http://127.0.0.1:8000/api/v1/sources?type=news&status=enabled"
```

### 3.5 编辑与停用

- **停用**：将 status 设为 `disabled`，该源不再参与手动/定时采集（界面实现后提供编辑表单）。
- **删除**：当前版本通过数据库维护；生产扩展可增加 DELETE 接口。
- **修改 keywords**：需重新编辑数据源配置；下次采集任务生效。

## 4 采集任务管理

### 4.1 手动启动采集

**适用场景：** 新增数据源后首次入库、演示前刷新数据、排查采集故障。

**界面操作：**

1. 进入「数据源管理」列表。
2. 找到目标数据源，点击「立即采集」。
3. 弹出任务进度提示，完成后显示 success_count / failed_count。

**API 操作：**

```powershell
curl -X POST http://127.0.0.1:8000/api/v1/tasks/collect ^
  -H "Content-Type: application/json" ^
  -d "{\"source_id\": 1}"
```

### 4.2 查询任务状态

```powershell
curl http://127.0.0.1:8000/api/v1/tasks/1
```

**表4-1 任务状态说明**

| task_status | 含义 | 管理员动作 |
|-------------|------|------------|
| running | 执行中 | 等待完成 |
| success | 至少 1 条入库成功 | 无需处理 |
| failed | 无有效条目入库 | 查看 error_message，检查 endpoint 与 keywords |

**响应字段：**

| 字段 | 说明 |
|------|------|
| success_count | 成功写入条数（去重后） |
| failed_count | 失败条数 |
| error_message | 错误摘要，多条以分号分隔 |
| started_at / finished_at | 任务起止时间 |

### 4.3 采集失败排查

**表4-2 故障排查流程**

| 步骤 | 操作 |
|:----:|------|
| 1 | 查看 `error_message` 字段 |
| 2 | 对照《数据源接入规范》检查 endpoint 格式 |
| 3 | 尝试放宽 keywords 或留空（测试环境） |
| 4 | 对 csv/api 类型确认样例文件存在 |
| 5 | 重新执行采集任务 |
| 6 | 仍失败则查看 audit_logs 并联系采集模块负责人 |

### 4.4 定时采集（扩展）

schedule 字段支持 hourly / daily / weekly。当前版本以手动触发为主；接入定时调度器（如 APScheduler）后，管理员在界面选择 schedule 即可自动执行。

## 6 审计日志

### 6.1 审计范围

系统自动记录以下操作至 `audit_logs` 表：

**表6-1 审计动作类型**

| action | 触发场景 |
|--------|----------|
| create_source | 新增数据源 |
| start_collect_task | 启动采集任务 |
| create_report | 生成报表 |

### 6.2 查看审计日志

**界面操作（规划）：**

1. 管理后台 → 「审计日志」。
2. 按时间范围、动作类型、操作对象筛选。
3. 点击单条记录查看详情（操作人、目标类型、目标 ID、时间）。

**数据库查询（当前可用）：**

```sql
SELECT id, action, target_type, target_id, created_at
FROM audit_logs
ORDER BY id DESC
LIMIT 50;
```

可使用 SQLite 客户端打开 `data/app.db` 执行。

### 6.3 审计日志字段

| 字段 | 说明 |
|------|------|
| user_id | 操作用户 ID（扩展登录后填充） |
| action | 操作类型 |
| target_type | 目标表名，如 data_sources |
| target_id | 目标记录 ID |
| created_at | 操作时间 UTC |

### 6.4 合规要求

- 审计日志采用**追加写入**，不允许通过业务界面修改或删除。
- 答辩或验收时可导出 audit_logs 作为「系统具备基本治理能力」的证明。

## 7 报表文件维护

### 7.1 生成报表

当前版本支持生成真实 XLSX 和 PDF 文件：

```powershell
curl -X POST http://127.0.0.1:8000/api/v1/reports ^
  -H "Content-Type: application/json" ^
  -d "{\"report_type\":\"summary\",\"format\":\"xlsx\",\"generated_by\":\"罗元恒\"}"
```

| 字段 | 可选值 | 说明 |
|------|--------|------|
| report_type | summary / detail | 摘要报表或明细报表 |
| format | xlsx / pdf | 文件格式 |
| generated_by | 任意文本 | 生成人标识，默认罗元恒 |

### 7.2 查询与下载

```powershell
curl http://127.0.0.1:8000/api/v1/reports
curl -O http://127.0.0.1:8000/api/v1/reports/1/download
```

报表文件默认保存到 `data/reports/`。文件名使用安全时间戳，避免 Windows 文件名非法字符。

### 7.3 报表备份

备份数据库时同步备份 `data/reports/`。如果只提交课程源码包，不提交运行期数据库和历史报表文件。

## 8 数据维护

### 8.1 数据库位置

默认路径：`data/app.db`（SQLite）。

### 8.2 备份

**操作步骤：**

1. 停止服务（或确保无写入操作）。
2. 复制 `data/app.db` 至备份目录，如 `backup/app-20260623.db`。
3. 同时备份 `data/reports/` 目录（如有报表文件）。

### 8.3 恢复

1. 停止服务。
2. 用备份文件覆盖 `data/app.db`。
3. 重新启动服务并执行 health 检查。

### 8.4 重置演示数据

删除 `data/app.db` 后重启服务，系统将重新 initialize 并 seed 演示数据源。

## 10 常见问题

| 问题 | 处理 |
|------|------|
| 端口被占用 | 更换 --port 或结束占用进程 |
| 数据库 locked | 确保只有一个服务实例写库 |
| 采集 success_count=0（第二次） | 正常，raw_hash 去重导致 |
| API 返回 400 | 检查 JSON、ID、limit、report_type 或 format 参数 |
| API 返回 404 | 检查路径与 source_id/item_id/report_id 是否存在 |
| API 返回 500 | 查看终端错误输出，检查数据库完整性和依赖安装 |
